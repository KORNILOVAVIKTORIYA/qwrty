# DriverMatching — подбор ближайших водителей на заказ

Реализация механизма подбора водителей на заказ на прямоугольной карте N×M.
В каждой ячейке карты может находиться не более одного водителя; у каждого
водителя есть уникальный идентификатор, задаваемый при добавлении/обновлении
координат. Реализовано **4 алгоритма** поиска 5 ближайших к заказу водителей,
все они покрыты тестами NUnit, производительность сравнивается с помощью
BenchmarkDotNet.

## Структура решения

```
DriverMatching.sln
├── src/DriverMatching.Core/              # библиотека с доменной моделью и алгоритмами
│   ├── Models/
│   │   ├── Location.cs                   # координата (X, Y) на карте
│   │   └── Driver.cs                     # водитель: Id + текущее Location
│   ├── Exceptions/
│   │   └── DomainExceptions.cs           # OutOfMapBoundsException, CellOccupiedException, DriverNotFoundException
│   ├── DriverRegistry.cs                 # хранилище водителей на карте Width×Height
│   └── Algorithms/
│       ├── INearestDriversFinder.cs      # общий контракт для всех алгоритмов
│       ├── DriverDistance.cs             # результат поиска: Id, координата, расстояние
│       ├── DistanceUtils.cs              # общая метрика расстояния (Евклидова)
│       ├── BruteForceSortFinder.cs       # алгоритм 1
│       ├── BruteForceHeapFinder.cs       # алгоритм 2
│       ├── GridRingSearchFinder.cs       # алгоритм 3
│       └── KdTreeFinder.cs               # алгоритм 4
├── tests/DriverMatching.Tests/           # NUnit-тесты
│   ├── DriverRegistryTests.cs            # тесты хранилища водителей
│   └── NearestDriversFinderTests.cs      # один набор тестов для всех 4 алгоритмов
└── benchmarks/DriverMatching.Benchmarks/ # консольное приложение BenchmarkDotNet
    ├── NearestDriversBenchmarks.cs
    └── Program.cs
```

## Доменная модель

- **`Location`** — координата ячейки (`X`, `Y`), `readonly record struct`.
- **`Driver`** — водитель с уникальным `Id` и текущим `Location`.
- **`DriverRegistry`** — хранилище водителей на карте `Width × Height`. Использует
  `Dictionary<int, Driver>` (по Id) и `Dictionary<Location, int>` (по координате),
  что позволяет работать с очень большими картами (например 100 000 × 100 000),
  не выделяя память под все ячейки сразу — память расходуется только на реально
  занятые ячейки.
  - `AddOrUpdateDriver(id, x, y)` — добавляет нового водителя или переносит
    существующего на новые координаты (старая ячейка освобождается).
    Бросает `OutOfMapBoundsException`, если координаты вне карты, и
    `CellOccupiedException`, если ячейка уже занята **другим** водителем.
  - `RemoveDriver(id)`, `GetDriver(id)` (бросает `DriverNotFoundException`,
    если не найден), `TryGetDriverAt(location, out id)`.

**Допущение по метрике расстояния.** В задании не указано явно, какое расстояние
считать «ближайшим». Реализация использует **евклидову метрику** — она
интуитивно ближе к понятию «ближайший водитель» на карте города (если бы
требовалось манхэттенское расстояние из-за движения только по дорогам сетки,
все 4 алгоритма легко переписываются под него, так как метрика инкапсулирована
в `DistanceUtils`).

## Алгоритмы поиска ближайших водителей

Все четыре реализуют общий интерфейс `INearestDriversFinder.FindNearest(registry, order, count)`,
возвращающий не более `count` водителей, отсортированных по возрастанию расстояния.

### 1. BruteForceSortFinder — «грубая сила» с полной сортировкой
Считает расстояние от заказа до **каждого** водителя, сортирует весь список и
берёт первые `count`. Самая простая и надёжная реализация — используется как
эталон («ground truth») для проверки корректности остальных трёх алгоритмов в тестах.

**Сложность:** O(D log D), где D — общее число водителей. Память: O(D).
Не зависит от размеров карты и взаимного расположения водителей.

### 2. BruteForceHeapFinder — перебор с ограниченной кучей
Тоже перебирает всех водителей, но вместо полной сортировки поддерживает
`PriorityQueue` (кучу) ограниченного размера `count`: на вершине всегда
«самый далёкий» из текущих топ-`count` кандидатов, который выбрасывается,
если находится более близкий водитель.

**Сложность:** O(D log count). Поскольку в задании `count = 5` — это
фактически O(D), то есть заметно быстрее полной сортировки при больших D.

### 3. GridRingSearchFinder — расширяющиеся кольца вокруг заказа
Использует то, что карта — именно сетка: вместо перебора всех водителей
проверяются по очереди границы (кольца) квадратов Чебышёвского радиуса
0, 1, 2, ... вокруг точки заказа через `DriverRegistry.TryGetDriverAt`.
Минимально возможное евклидово расстояние от заказа до любой ячейки на
кольце радиуса r равно r, поэтому, как только среди найденных кандидатов
набралось `count` штук и расстояние до самого далёкого из них ≤ r — дальше
расширять поиск бессмысленно.

**Сложность:** существенно зависит от плотности водителей рядом с заказом.
Если водители расположены близко — алгоритм находит ответ почти мгновенно,
проверив лишь малую часть карты (≈O(r²) для итогового радиуса r). Если же
водителей мало и они далеко (либо карта огромна, а водителей единицы) —
в худшем случае придётся просканировать кольца почти до краёв карты, то
есть до O(N×M).

### 4. KdTreeFinder — k-d дерево (k = 2)
Строит сбалансированное k-d дерево по координатам всех водителей и обходит
его, отсекая целые поддеревья, которые заведомо не могут содержать водителя
ближе уже найденных `count` кандидатов (стандартный nearest-neighbour поиск
по k-d дереву).

**Важное архитектурное решение:** дерево строится **заново при каждом вызове**
`FindNearest`, а не хранится и обновляется инкрементально между запросами.
Это осознанный выбор: по условию задачи водители часто меняют координаты
(`AddOrUpdateDriver` может вызываться произвольно часто), а полноценная
инкрементальная балансировка k-d дерева при частых перемещениях — отдельная,
значительно более сложная задача, выходящая за рамки этого задания.

**Сложность:** построение дерева — O(D log D) на каждый вызов; сам обход —
в среднем O(log D + count), в худшем случае (вырожденное дерево) — O(D).
Из-за перестройки на каждый запрос итоговая сложность вызова в целом близка
к O(D log D) — то есть теоретическое преимущество k-d дерева (быстрый поиск
после однократного построения) здесь не реализуется в полной мере именно
из-за частых перемещений водителей. Это специально показано и объяснено
в разделе с результатами бенчмарков ниже.

## Тестирование (NUnit)

Тесты находятся в `tests/DriverMatching.Tests`:

- **`DriverRegistryTests`** — CRUD-операции хранилища: добавление/перемещение
  водителя, освобождение старой ячейки, исключения при выходе за границы карты
  и при занятой ячейке, поиск/удаление несуществующего водителя.
- **`NearestDriversFinderTests<TFinder>`** — один параметризованный набор
  тестов (`[TestFixture(typeof(...))]`), который прогоняется **для каждого
  из 4 алгоритмов**: пустой реестр, водителей меньше чем `count`, `count = 0`,
  заказ точно на ячейке водителя (расстояние 0), сортировка результата по
  возрастанию расстояния, ровно `count` результатов при избытке водителей,
  водители по углам карты, карта шириной в одну ячейку, а также **кросс-валидация
  на 25 случайных сценариях** против эталонного `BruteForceSortFinder` (сравнение
  расстояний с округлением, а не идентификаторов водителей — чтобы не получать
  ложных ошибок при совпадающих расстояниях у разных водителей).

Запуск тестов:

```bash
dotnet test
```

> **Важно:** в песочнице, где собиралось это решение, исходящий доступ к
> `api.nuget.org` заблокирован, поэтому restore пакетов NUnit (и
> BenchmarkDotNet) здесь невозможен — `dotnet test` нужно запускать на вашей
> машине с обычным доступом к NuGet. Сама библиотека `DriverMatching.Core`
> не имеет внешних зависимостей и собирается без проблем в любой среде; вся
> логика всех 4 алгоритмов и реестра была дополнительно вручную перепроверена
> отдельным консольным скриптом (50 проверок, включая те же 25 сценариев
> кросс-валидации, что и в финальных NUnit-тестах) — все проверки прошли
> успешно, так что результаты `dotnet test` на вашей машине ожидаются зелёными.

## Бенчмарки (BenchmarkDotNet)

Находятся в `benchmarks/DriverMatching.Benchmarks/NearestDriversBenchmarks.cs`.
Сравниваются все 4 алгоритма на карте 1000×1000 при числе водителей
100 / 1 000 / 10 000 / 50 000, точка заказа — в центре карты, искомое
количество водителей — 5 (как в задании). Используется `[MemoryDiagnoser]`
для дополнительного сравнения по памяти.

Запуск (обязательно в конфигурации **Release** — иначе BenchmarkDotNet
откажется запускаться):

```bash
dotnet run -c Release --project benchmarks/DriverMatching.Benchmarks
```

После завершения BenchmarkDotNet выведет таблицу результатов в консоль и
сохранит более подробный отчёт (markdown/html) в
`benchmarks/DriverMatching.Benchmarks/BenchmarkDotNet.Artifacts/results/`.

```

```

**Предварительные ориентировочные замеры (не замена официальному отчёту
BenchmarkDotNet).** Чтобы проверить ожидаемое поведение алгоритмов ещё на
этапе разработки, был выполнен черновой замер через `Stopwatch` (200
итераций на сценарий, 5 прогонов на разогрев) с теми же параметрами —
карта 1000×1000, заказ в центре, среднее время одного вызова в микросекундах:

| Кол-во водителей | BruteForceSort | BruteForceHeap | GridRingSearch | KdTree |
|---:|---:|---:|---:|---:|
| 100 | 43.9 | 12.7 | 4247.9 | 239.0 |
| 1 000 | 560.9 | 114.8 | 755.4 | 2978.7 |
| 10 000 | 5503.9 | 242.9 | 125.0 | 16288.8 |
| 50 000 | 9193.6 | 414.5 | 13.1 | 63138.8 |

Это **не официальный отчёт** (нет warm-up по методологии BenchmarkDotNet,
нет статистики по доверительным интервалам и т.д.), но видна ожидаемая
картина, которую должен подтвердить и официальный бенчмарк:

- **BruteForceSort** линейно деградирует с ростом числа водителей — ожидаемо
  для O(D log D).
- **BruteForceHeap** стабильно один из самых быстрых вариантов на любой
  плотности — хороший «дефолтный» выбор именно из-за частых перемещений
  водителей в этой задаче.
- **GridRingSearch** крайне медленный при малом числе водителей (приходится
  сканировать огромный пустой радиус вокруг заказа), но становится почти
  мгновенным при высокой плотности водителей — расстояние до ближайших
  находится за пару маленьких колец.
- **KdTree** здесь оказывается самым медленным при росте D — но это прямое
  следствие того, что дерево перестраивается с нуля при каждом вызове
  (см. раздел про алгоритм 4 выше): k-d деревья выигрывают, когда строятся
  один раз и потом обслуживают много запросов между обновлениями, а в этой
  задаче с часто двигающимися водителями цена перестройки доминирует.
